Deep dive technical analysis

nanoboss

A procedure-oriented agent orchestration system built on a deliberately mixed transport stack: HTTP/SSE for frontends, ACP over stdio for downstream agents, and a globally registered MCP server for durable session inspection and slash-command dispatch.
nanoboss is not just “a CLI for an LLM.” It is a compact agent runtime with durable provenance, pluggable procedures, session-aware multi-agent composition, and an unusually strong emphasis on inspectable state.

2 · mental model

nanoboss is a procedure runtime with chat UX, not a chat UI with a few commands

Core abstraction stack

FrontendTUI CLI and HTTP clients consume event streams and submit prompts.
→
ServiceNanobossService owns sessions, active runs, cancellation, replay, continuations, and prompt assembly.
→
ProceduresSlash commands are typed, composable procedures that can call agents and other procedures.
Downstream agentsClaude, Gemini, Codex, Copilot via ACP over stdio.
↔
Global MCPOne nanoboss MCP server exposes durable session inspection and async dispatch tools.
↔
Durable storeRuns, refs, session metadata, replay events, and repo-local workflow artifacts.

Why this matters

  • Plain text is just the default command path. The real product surface is the procedure system.
  • State is first-class. The system persists intermediate work as inspectable runs/refs rather than relying only on model memory.
  • Composition is explicit. Procedures can nest, share or isolate conversation state, and recover async dispatches.
  • Frontends are thin. The TUI is sophisticated, but the architectural center of gravity is still the service + session store.
hidden /default plain-text fallback 24 built-in procedure files synthetic /dismiss continuation command workspace/profile disk procedures
This is the right mental model for reading the repo: service + procedures + durable state, then transports/UI.

4 · durable state

Durable provenance is one of nanoboss’s strongest architectural decisions

State topology

~/.nanoboss/ sessions/<sessionId>/ session.json run records/ <timestamp>-<runId>.json procedure-dispatch-jobs/ procedure-dispatch-progress/ run kinds: top_level → user-visible command runs procedure → nested procedure calls agent → downstream agent invocations run.output may contain: data display stream summary memory pause replayEvents explicitDataSchema
  • Refs point into durable run payloads, so procedures and MCP tools can pass precise references instead of copying blobs.
  • Replay events let the TUI restore prior runs with transcript/tool-card fidelity.
  • Session metadata is canonical in session.json; current-sessions.json is a workspace cache used for lookup/resume.

Why it is architecturally important

  • It breaks the usual “agent state only exists in chat context” trap.
  • It enables post-hoc provenance: agents can answer “how did you reach that?” by querying durable runs/refs over MCP.
  • It supports nested composition without overloading the parent chat transcript.
  • It cleanly separates discovery summaries (memory cards) from authoritative stored state (runs/refs).
This is the repository’s sharpest systems idea: durable, inspectable state is treated as a product feature, not just an implementation detail.
memory cards for compact recall MCP tools for exact retrieval recovery from async dispatch timeouts
nanoboss prefers MCP-based retrieval over filesystem spelunking and even injects guidance telling agents to do so.

5 · procedure platform

Procedures are the programmable substrate

What the platform supports

  • Built-ins: loaded lazily via deferred metadata entries in ProcedureRegistry.
  • Disk procedures: discovered recursively under repo-local and profile roots with no manifest file.
  • Generated procedures: /create prompts an agent to emit TypeScript, persists it, then registers it live.
  • Typed outputs: procedures are encouraged to use typia + jsonType(...) for schema-backed agent results.

Visible built-in families

general: create, model, tokens, research, linter, simplify, simplify2, second-opinion kb/*: ingest, compile-source, compile-concepts, link, render, health, refresh, answer autoresearch/*: start, continue, status, clear, finalize nanoboss/*: commit, pre-commit-checks

Most unusual implementation detail

  • Disk-procedure metadata is inferred by static source parsing: name, description, inputHint, executionMode.
  • Actual module loading happens later via Bun.build(...) with a typia plugin and a content-hash build cache.
  • If the target workspace lacks the required runtime bits, nanoboss temporarily symlinks runtime node_modules and src/ into place.
Strength: very low ceremony for extensibility. Risk: the loader path is powerful but non-trivial, and the static metadata inference is intentionally heuristic.
Registry hot spots - src/procedure/registry.ts 27 outbound imports - src/procedure/disk-loader.ts handles discovery, build cache, temp symlinks, persistence routing - docs/procedure-packages.md explicitly avoids manifests
This is a “procedures as code modules” system, not a prompt-template registry.
Evidence: src/procedure/registry.ts · src/procedure/disk-loader.ts · src/procedure/create.ts · docs/procedure-packages.md
6 · downstream session semantics

The fresh/default/inherit split is subtle, but it is one of nanoboss’s best API ideas

Conversation-binding model

Call shapeDefaultMeaning
ctx.agent.run(...)freshnew isolated ACP session
ctx.agent.run(..., { session: "default" })explicitreuse the session’s persistent master conversation
ctx.procedures.run(name, prompt)inheritchild procedure keeps caller’s default-binding behavior
ctx.procedures.run(..., { session: "fresh" })explicitchild gets a private default conversation
ctx.procedures.run(..., { session: "default" })explicitchild rebinds to the master session
This lets nanoboss separate workflow state from conversation state. That is a big deal.

How flagship procedures use it

  • /default is just ctx.agent.run(prompt, { session: "default" }) — persistent chat continuity.
  • /research creates a brief in the default conversation, then dispatches a fresh isolated research worker.
  • /simplify2 intentionally keeps most direct agent calls fresh so each analysis phase is isolated and typed.
  • /second-opinion gets a first pass from the default model, then a fresh typed critique from Codex using refs.
Why this matters: continuity where it helps + deterministic isolated substeps where it matters + typed JSON parse/retry on both paths = fewer “chat memory accidentally contaminated the workflow” failures
There is real design depth here: nanoboss makes session selection an explicit part of the programming model.
Evidence: src/core/types.ts · src/core/context.ts · src/agent/default-session.ts · procedures/default.ts · procedures/research.ts · docs/simplify2.md

Architectural read

  • Good: the CLI runs against the same HTTP/SSE API as remote clients; private mode is just deployment, not a separate code path.
  • Risk: TUI complexity is rising fast. src/tui/ is 4.5k LOC, and reducer.ts alone is 949 LOC.

The product experience is clearly important, but TUI interaction logic is becoming one of the repo’s major complexity centers.
Private-server mode is a particularly smart choice: it keeps UX friendly while preserving one service contract.

Knowledge-base pipeline that ingests raw files, compiles summaries/concepts, links pages, renders outputs, and writes manifests/indexes under repo-local state.

Experiment-loop surface for optimization sessions with benchmark commands, logs, and keep/discard control flow.

  • stateful experiment runner
  • foreground loop orchestration
  • ties into repo workflow
The repo proves nanoboss is a workflow platform. The interesting question is not “can it chat?” but “how many structured agent workflows can be expressed cleanly before orchestration complexity wins?”

9 · build, distribution, operations

Packaging and operator ergonomics are unusually mature for a project this size

Distribution choices

  • build.ts uses Bun.build(... compile ...) to emit a standalone dist/nanoboss binary.
  • The build embeds the git commit into the runtime banner and into server/CLI health.
  • Build logic also estimates bundle/runtime size attribution and installs runtime assets for disk procedures into ~/.nanoboss/runtime.
  • The CLI can warn when the installed binary appears stale relative to build-relevant files in the source tree.
binary size observed: ~70 MB build freshness detection workspace fingerprinting

Operational controls

  • doctor --register repairs global MCP registration for Claude, Codex, Gemini, and Copilot.
  • Server health checks include workspace identity and procedure fingerprint so external CLI connections can reject incompatible servers.
  • Agent transcripts are logged under ~/.nanoboss/agent-logs, while direct recursive access to that area is explicitly blocked.
  • Token metrics are enriched from ACP usage updates, prompt responses, and provider-specific side channels.
This is a strong “toolsmith” codebase: the developer/operator experience has been considered as a first-class feature.
Operational maturity is ahead of what the small dependency footprint would suggest.
Evidence: build.ts · src/core/build-freshness.ts · src/core/doctor.ts · src/http/server-supervisor.ts · src/agent/token-metrics.ts

11 · complexity hotspots

Where the design is excellent — and where it is starting to bend

Architectural strengths

  • Durable provenance model is better than most agent tools.
  • Explicit session-binding semantics avoid a lot of hidden-chat-state bugs.
  • Procedure composition gives the system a real programming model.
  • Shared service contract across private CLI and remote clients keeps architecture honest.
  • Operator ergonomics — doctor, build freshness, server compatibility, cached pre-commit checks — are excellent.

Primary risks

  • NanobossService is overloaded. It manages session lifecycle, event logging, prompt augmentation, continuation state, dispatch bridging, cancellation, recovery, and command projection.
  • /simplify2 is a product inside the product. It is powerful, but the sheer size means conceptual drift can hide inside a single file.
  • TUI state complexity is rising. Focus-picker and checkpoint flows are now spanning core types, repository serialization, controller logic, app overlays, and tests.
  • Disk procedure loading is clever but fragile. Static metadata inference + runtime compilation + temp symlinks is a lot of machinery for one subsystem.

12 · recommendations

If I were steering nanoboss next, I would optimize for conceptual compression

1. Decompose the service core

  • extract prompt-composition / memory-card logic
  • extract continuation manager
  • extract async dispatch/recovery coordinator

Goal: keep NanobossService as a session façade, not a mega-orchestrator.

2. Split simplify2 by concern

  • focus state & persistence
  • analysis/ranking pipeline
  • apply/validate/commit path
  • continuation UI mapping

Goal: preserve the feature while making it legible enough to keep evolving.

3. Harden the procedure boundary

  • formalize pre-load metadata vs loaded procedure types
  • consider a cached descriptor index for disk procedures
  • reduce regex-only metadata inference over time

Goal: keep extensibility without making loader behavior too magical.

4. Keep dogfooding — but with stronger seams

  • The repo-local .nanoboss/ artifacts and simplify2 journals are evidence that nanoboss is already improving itself.
  • That is a strategic advantage; keep it. But push more of that capability behind smaller, separately testable library boundaries.

5. Add one canonical “state topology” doc

  • global home state under ~/.nanoboss
  • repo-local workflow state under .nanoboss and .kb
  • protocol paths: HTTP/SSE, ACP stdio, MCP stdio

The architecture is strong enough that it deserves a first-class map.

My advice is evolutionary, not corrective: preserve the architecture, shrink the conceptual blast radius.

13 · verdict

Overall assessment

nanoboss is an unusually thoughtful agent harness: it combines durable provenance, composable typed procedures, multi-agent orchestration, and a unified frontend/service contract in a codebase that is still small enough to reshape.

What is best in class

  • stateful provenance via runs/refs
  • session-binding API design
  • workflow-oriented procedure surface
  • operator ergonomics

What will determine success

  • whether service/simplify2/TUI complexity keeps decomposing
  • whether procedure extensibility stays simple as the platform matures
  • whether durable state remains the architectural center

Bottom line

This is a serious systems design, not a thin wrapper. Its main risk is not lack of ambition — it is carrying too much successful ambition in too few files.
