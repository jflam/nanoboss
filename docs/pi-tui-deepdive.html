1 · integration seam

Nanoboss deliberately treats pi-tui as a toolkit, not as an app framework

The seam

src/tui/pi-tui.ts export { TUI, ProcessTerminal, Editor, Container, Text, TruncatedText, Markdown, Box, SelectList, Spacer, CombinedAutocompleteProvider, matchesKey, isKeyRelease, ... } from "@mariozechner/pi-tui";
  • Nanoboss has a single local adapter file for all pi-tui imports.
  • That keeps upstream churn isolated and makes local migration/version pinning easier.
  • It also makes clear that nanoboss is consuming public pi-tui APIs only.

What nanoboss actually uses

Categorypi-tui APIs used
RuntimeTUI, ProcessTerminal
InputEditor, CombinedAutocompleteProvider, matchesKey, isKeyRelease
LayoutContainer, Spacer, Box
RenderingText, TruncatedText, Markdown, SelectList
Notably absent from the main app path: Input, Loader, SettingsList, Image, and pi-tui’s real floating overlay stack.
The seam is thin by design.

2 · construction path

The main app is built as controller + view on top of pi-tui primitives

Instantiation flow in src/tui/app.ts

step 1
new ProcessTerminal()
↓
step 2
new TUI(terminal, false)
↓
step 3
new Editor(tui, theme.editor, { paddingX: 1, autocompleteMaxVisible: 8 })
↓
step 4
new NanobossTuiController(...)
↓
step 5
new NanobossAppView(editor, theme, state)

Responsibility split

  • Controller owns HTTP/SSE session lifecycle, event handling, prompt forwarding, cancellation, and pending prompts.
  • View owns header/footer/transcript/composer composition.
  • pi-tui owns terminal lifecycle, focus, editor mechanics, text rendering, and repaint behavior.
This is a classic “app state on top of a rendering runtime” architecture. Nanoboss is not trying to fork or shadow pi-tui internals.
Nanoboss uses pi-tui as a substrate, not as a source of business logic.
Evidence: src/tui/app.ts · src/tui/controller.ts · src/tui/run.ts
3 · UI topology

The root view is an append-friendly vertical document

Root layout in src/tui/views.ts

Container ├─ header (TruncatedText) ├─ session line (TruncatedText) ├─ status line (TruncatedText) ├─ Spacer ├─ transcript (custom component) ├─ composer container │ └─ Editor or inline selector/continuation component ├─ Spacer ├─ activity bar (TruncatedText) └─ footer/help line (TruncatedText)
  • This layout is very native to pi-tui’s model: each piece renders to lines.
  • TruncatedText is used for single-line bars; Markdown/Text are used for body content.

Transcript strategy

  • TranscriptComponent keeps stable component instances for turns and tool calls.
  • It rebuilds structure only when ordering changes; otherwise it mutates existing components.
  • That aligns well with pi-tui’s line-diff renderer, which performs best when updates are append-mostly and localized.
stable identity maps incremental updates cheap footer/header bars edits above viewport are expensive
Nanoboss’s persistent transcript model is cooperating with pi-tui, not fighting it.

4 · local UI patterns

Nanoboss builds its own cards, and its “overlays” are mostly composer swaps

Cards are local components

  • MessageCardComponent and ToolCardComponent are nanoboss code.
  • They are composed from Box + Text + Container.
  • So visual semantics live in nanoboss, while spacing/background/wrapping mechanics come from pi-tui.
message-card.ts / tool-card.ts → Box(1, 1, backgroundFn) → Text(lines.join("\n"))

“Overlay” nuance

  • Files under src/tui/overlays/ use pi-tui components like SelectList.
  • But the main app does not use TUI.showOverlay() for them.
  • Instead, nanoboss swaps the composer area with view.showComposer(component) and focuses that component.
This means the simplify/model pickers are inline modal-ish composer replacements, not true composited floating overlays from upstream pi-tui.
Good to remember when comparing nanoboss UX to upstream pi apps.
Evidence: src/tui/components/*.ts · src/tui/overlays/*.ts · src/tui/app.ts
5 · upstream runtime

pi-tui is fundamentally a diffing line renderer over a terminal component tree

Core model in ~/src/pi-mono/packages/tui/src/tui.ts

Component { render(width): string[] handleInput?(data) wantsKeyRelease? invalidate() } TUI extends Container → render full tree to newLines → compare with previousLines → patch changed region if possible → full redraw when necessary
  • pi-tui is not virtual DOM and not React-style reconciliation.
  • Its unit of truth is rendered terminal lines.
  • It uses synchronized output to reduce flicker during writes.

Why redraws happen

  • Always full redraw: first render, width change.
  • Usually full redraw: height change, content change above viewport, shrink-cleanup cases.
  • Cheap path: localized changes at or below visible viewport, especially append-mostly updates.
This is the key reason transcript architecture matters. If nanoboss mutates lines above the viewport, pi-tui may need to repaint far more aggressively.
Many observed scroll/jump/perf behaviors ultimately reduce to this file.
Evidence: ~/src/pi-mono/packages/tui/src/tui.ts
6 · input path

Nanoboss inherits a surprisingly deep terminal/input/editor stack from pi-tui

Terminal layer

  • ProcessTerminal enables raw mode, bracketed paste, resize handling, title updates.
  • Queries Kitty keyboard protocol and falls back to xterm modifyOtherKeys.
  • Drains stdin on shutdown to avoid leaked key-release events.

Key normalization

  • keys.ts normalizes legacy escapes, Kitty CSI-u, non-Latin layouts, repeats/releases.
  • stdin-buffer.ts protects against partial/batched escape sequences.
  • keybindings.ts centralizes named actions.

Editor layer

  • Multiline wrap + vertical scroll
  • history + undo + kill ring
  • jump-to-char + word motions
  • paste markers + async autocomplete
  • IME hardware cursor support

What nanoboss adds on top

  • Global input interception via tui.addInputListener(...) for busy-state behavior: steer on Enter, queue on Tab, stop on Esc, toggle tool output on Ctrl+O, toggle simplify auto-approve on Ctrl+G.
  • A subclassed autocomplete provider to preserve leading slash behavior for namespaced commands like /autoresearch/clear.
If keyboard behavior feels weird, upstream is usually the first suspect.
Evidence: terminal.ts · keys.ts · stdin-buffer.ts · components/editor.ts · src/tui/app.ts

8 · verdict

The integration is good; the main strategic risk is version drift around exactly the behaviors nanoboss cares about

Version note

  • Nanoboss depends on @mariozechner/pi-tui 0.65.2.
  • The monorepo source currently reports 0.66.1.
  • The changelog between those versions includes fixes for heavy streaming render scheduling, long-session container behavior, overlay correctness, and autocomplete debounce.
That means the nanoboss/pi-mono comparison should always be done with version awareness. The source tree you inspect may be slightly ahead of what nanoboss is actually running.

Final assessment

Nanoboss uses pi-tui in the right way: small public seam, app-owned semantics, toolkit-owned runtime. Most future gains will come from staying aligned with upstream rendering/input/editor improvements rather than forking the stack.
