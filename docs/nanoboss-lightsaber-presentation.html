1 · thesis

One-size-fits-all agent harnesses optimize for demoability, not for developer workflow reality

What a generic harness usually assumes

One memory model

Usually a blob of chat history, maybe some vector retrieval, rarely a durable execution tree with refs.

One control loop

Prompt, tools, answer. That is fine for chat, weak for research loops, validation gates, or resumable workflows.

One continuity policy

Either always fresh or always persistent. Real systems need both, often inside the same procedure.

One success criterion

Usually “got an answer.” Developers often need “produced artifacts, passed checks, left provenance.”

What nanoboss is trying instead

Treat workflow logic as code, not as prompt folklore. Generic harnesses usually remove friction at exactly the wrong moment.
  • Procedures encode domain-specific orchestration in TypeScript.
  • Each procedure chooses its own agent continuity, structure, and validation path.
  • Runs are durable, traversable, and ref-addressable after execution.
  • Agents can inspect that durable state through a stable MCP surface.
  • The runtime is meant to be extended by developers, not merely configured by toggles.

2 · psychology

Cheap code makes it easier to skip judgment

“The moments where you want to skip thinking are exactly the moments where thinking matters most.”

What changed

  • The old bottleneck was typing and implementation throughput.
  • The new bottleneck is deciding when not to generate code yet.
  • When code becomes cheap, premature implementation becomes psychologically attractive.
  • That temptation is strongest exactly when the problem is still underspecified.

What matters now

The core engineering skill is no longer just producing code. It is knowing where to slow the system down before speculative structure becomes real.

Skip Thinking Notes

  • Pause on the quote and read it slowly.
  • AI lowers the cost of acting, not the cost of being wrong.
  • The new failure mode is implementing before understanding.
  • Bridge: once the wrong code exists, the system starts defending it.
This is the behavioral failure mode behind bad agent workflows: cheap action outruns judgment.
Quote source: Armin Ronacher, AI Engineer Talk, slide 9
3 · psychology

Messy codebases become self-reinforcing under agents

Why entropy compounds

cheap code -> premature implementation -> messy local pattern -> agent sees existing pattern as precedent -> more surface area built on top -> deletion feels riskier -> even more code gets added around the mess
Agents are generally much better at extending local patterns than deleting them. Deletion requires global confidence, boundary reasoning, and risk ownership.

Why this matters for your codebase

  • Once bad code exists, every future agent run is biased toward preserving it.
  • Removing code is scary because it forces a stronger proof burden than adding it.
  • Agentic cleanup is expensive, uncertain, and often deferred.
  • The best time to remove bad code is before it exists.
That is why “skip thinking” is so dangerous. The wrong structure hardens faster than the organization can reason it back out.

Entropy Notes

  • Say explicitly that agents are better at adding than deleting.
  • Deletion feels scary because it needs global confidence, not local plausibility.
  • Your punchline: the best time to remove bad code is before it exists.
  • Bridge to human placement: where do we insert judgment before the mess hardens?
Cheap generation changes the economics of entropy: mess appears faster, deletion gets harder.

4 · psychology

The human belongs at structural checkpoints, not everywhere and not only at the end

Where judgment matters most

  • Before a vague idea becomes durable code.
  • Before a local patch establishes a precedent the next agent will copy.
  • Before a boundary crossing or debt decision gets hidden inside generated output.
  • Before deletion becomes harder than addition.

What the runtime should do

  • Allow fast agentic execution when the structure is already clear.
  • Insert deliberate friction when the next step changes architecture or complexity.
  • Persist evidence so a human can review the actual reasoning path, not just the final patch.
  • Make checkpointing a first-class workflow concept instead of an ad hoc habit.

Why this fits nanoboss

The question is not whether to keep a human in the loop. The question is where. Nanoboss procedures, continuations, and harness checkpoints are a way to encode those judgment gates directly into the workflow.

Judgment Gates Notes

  • This is still the psychology section, not the org-design section.
  • Make it concrete: the human matters before speculative structure becomes durable.
  • Transition: nanoboss matters because procedures let you encode those gates.
This is the workflow answer to cheap code: place human judgment exactly where structural mistakes become expensive.

5 · design principle

nanoboss is a harness construction kit: the runtime stays small, the workflows get specialized

Primitive
Session

The user-facing boundary. Owns cwd, event stream, default agent continuity, and durable history.

Primitive
Run

The durable execution node. Top-level runs can contain nested procedure and agent runs.

Primitive
Ref

The stable pointer into stored output data. Keeps the runtime ref-heavy instead of copy-heavy.

Primitive
Continuation

The pause-and-resume model for long workflows, checkpoints, and human steering.

Primitive
Procedure

The authoring abstraction. This is where developers build their lightsaber.

Why this matters

If the primitives are stable and inspectable, developers can build very different workflows on top of the same runtime: validation-gated commits, resumable architecture simplification, repo-local knowledge pipelines, bounded research loops, or model-selection procedures. The point is not just customization. The point is to encode where thinking, review, and pause points belong.

Principle Notes

  • This is the thesis slide for the technical half.
  • Stress that procedures are not plugin fluff, they are authored workflow semantics.
  • Use the phrase: the runtime stays small, the workflows get specific.
nanoboss is opinionated about the runtime shape, not about a single universal workflow or a single universal level of friction.

6 · transport stack

Three transport roles, one runtime

High-level flow

FrontendCLI or HTTP client submits prompts and renders live events.
→
NanobossServiceCreates sessions, routes commands, handles continuations, cancellation, replay, and dispatch.
→
Procedure runtimeRuns a procedure with ctx.agent, ctx.state, ctx.procedures, ctx.session, ctx.ui.
Downstream agentCodex, Copilot, Claude, Gemini, or another ACP-capable tool.
←
ACP over stdioUsed for agent invocation and default-session reuse.
→
Global nanoboss MCPLets agents inspect runs/refs and dispatch procedures through a stable tool plane.

Why the stack is mixed on purpose

ConcernProtocolReason
Frontend UXHTTP + SSENatural fit for remote UIs and event streaming.
Agent executionACP over stdioMatches downstream CLIs and persistent conversation loading.
Runtime inspectionMCP over stdioGives every agent the same durable tool surface for runs, refs, and dispatch.
This is a key point for the talk: nanoboss does not worship one protocol. It picks the best protocol for each role.

Architecture Notes

  • Explain the mixed stack as intentional specialization by role.
  • Users talk over HTTP/SSE or CLI, agents over ACP, runtime inspection over MCP.
  • The architecture mirrors the larger thesis: do not force one abstraction everywhere.
Mixed stack is a feature, not an architectural accident.

7 · durable state

The real runtime artifact is not the chat transcript. It is the session-backed run graph.

Execution tree

Session top-level run: /simplify2 child run: procedure /nanoboss/pre-commit-checks child run: agent call (structured JSON) child run: agent call (default conversation) top-level run: /kb/refresh top-level run: /second-opinion
Each node is persisted by SessionStore, with summaries, display text, stream text, refs, replay events, and parent linkage.

What this buys you

  • Agents and procedures can traverse ancestors and descendants, not just “the last message.”
  • Structured outputs can stay durable and small through refs instead of being recopied into prompts.
  • Async procedure dispatch can recover because the underlying run exists independently of one client session.
  • Continuations are first-class because pause state is stored with the run, not hidden in UI state.

8 · authoring surface

The procedure API is the lightsaber hilt: this is where developers shape workflow behavior

What a procedure gets

ctx.agent
Invoke a downstream agent fresh or on the session's default conversation, with typed JSON parsing, named refs, and explicit model selection.
ctx.procedures
Compose procedures as child runs while inheriting or overriding session continuity.
ctx.state
Traverse durable run history and read or materialize refs.
ctx.session
Inspect or change the default downstream agent selection and token usage.
ctx.ui
Stream progress, statuses, notices, and cards to the frontend while the procedure runs.

Why this beats a monolithic loop

export default { name: "my-workflow", async execute(prompt, ctx) { const brief = await ctx.agent.run(prompt, { session: "default" }); const analysis = await ctx.agent.run( "return structured JSON", ResultType, { session: "fresh", refs: { brief: brief.dataRef } } ); const prior = await ctx.state.runs.topLevelRuns({ procedure: "my-workflow", limit: 3 }); ctx.ui.status({ phase: "analysis", message: "Validated plan, ready for next step" }); return { data: analysis.dataRef, summary: "custom workflow completed" }; } }
Fresh vs default vs inherit is not a minor option. It is how the author chooses when continuity helps, when isolation matters, and where a human should be brought back into the loop before more code is generated.

Procedure API Notes

  • This is the lightsaber hilt slide. Slow down here.
  • Walk through ctx.agent, ctx.procedures, ctx.state, ctx.session, ctx.ui as judgment tools.
  • Fresh vs default vs inherit is where authors decide continuity and risk boundaries.
Procedures are deterministic workflow code wrapped around agentic subcalls and explicit judgment placement.

9 · control semantics

nanoboss does not just execute commands. It routes plain text, continuations, and harness modes deliberately.

Prompt routing

  • If input starts with /name, nanoboss dispatches that procedure.
  • If input is plain text and a procedure is paused, the reply goes back to that continuation.
  • Otherwise plain text flows to /default, the persistent master conversation.
That means the chat surface is only the shell. The real semantics are driven by procedure routing and continuation state. Continuations are not just UX. They are deliberate friction.

Execution modes in practice

ModeUse whenExample
defaultConversationConversation continuity is the point./default, brief-building, exploratory dialogue
harnessThe procedure owns control flow and may call agents internally./simplify2, /model, dismiss continuation
fresh subcallsNeed isolation, deterministic structure, or validation passes.typed JSON extraction, lint analysis, one-shot research subtask

Routing Notes

  • Important line: continuations are deliberate friction.
  • This is the cleanest technical answer to the psychology slides.
  • Nanoboss can force the workflow to stop exactly where human judgment matters.
This is where nanoboss stops being "slash commands on top of chat" and becomes a control runtime with explicit checkpoint semantics.

10 · extension model

Custom workflows are loaded like software, not pasted like prompts

How procedures enter the system

Built-insImported directly in src/procedure/registry.ts.
→
Workspace disk procedures./.nanoboss/procedures for repo-specific workflows.
→
Profile procedures~/.nanoboss/procedures for personal cross-repo tools.
/create is important strategically: nanoboss treats procedure authoring as a runtime capability, not a side quest.

What makes the supply chain feel like software

  • Disk procedures are discovered recursively by static metadata, no manifest required.
  • Procedure modules are compiled with Bun, cached by source-graph hash, and lazy-loaded.
  • Helpers can live alongside entrypoints, so workflows can become packages, not giant single files.
  • The repo and profile procedure roots make custom workflows naturally local or reusable.

The extensibility story is code-first, file-backed, and versionable.

11 · concrete workflows

Different procedures embody different theories of agent work

/simplify2

A large harness procedure with observations, hypotheses, checkpoints, apply/reconcile phases, and resumable human decisions.

/autoresearch/*

A package of repo-local optimization commands with durable state, status, continue, clear, and finalize semantics.

/kb/*

A compiled knowledge workflow: ingest, compile source pages, compile concepts, link structure, render outputs, answer questions.

/nanoboss/commit

A validation-gated commit workflow layered on top of pre-commit checks and agent-generated commit context.

Same runtime, different harness semantics

  • /simplify2 is a hypothesis-driven architecture loop with explicit checkpoints before more complexity is allowed into the codebase.
  • /autoresearch is a bounded optimization process with explicit lifecycle commands.
  • /kb/refresh is a multi-stage data pipeline.
  • /second-opinion is a multi-model critique composition.

Talk line to emphasize

These are not "the same agent with different prompts." They are different workflow machines built on a shared runtime.

Workflow Examples Notes

  • Use this slide to prove these are distinct workflow machines, not one agent with different prompts.
  • Spend most time on simplify2 because it best matches the checkpoint and entropy thesis.
  • Then gesture quickly at autoresearch, KB, and commit as other workflow shapes.
This is the strongest evidence against one universal harness.

12 · agent-facing runtime

The downstream agent is not trapped in the prompt. It gets a controlled runtime via MCP.

What the agent can do through the global nanoboss MCP server

  • List procedures and inspect one procedure's metadata.
  • Start an async procedure dispatch and wait for completion.
  • List top-level or recent runs for a session.
  • Read exact runs, ancestors, descendants, refs, and schemas.
This is a powerful architectural move: agents can reason over durable workflow state without scraping files or relying on hidden prompt context.

Why that matters for custom harnesses

Agent session runtime mounts global nanoboss MCP -> agent calls procedure_dispatch_start -> nanoboss background worker executes procedure -> durable run is written to SessionStore -> agent calls procedure_dispatch_wait / get_run / read_ref
Nanoboss is effectively giving the model a stable API for the workflow runtime itself.

MCP Notes

  • The model is not trapped inside prompt text; it gets a runtime API.
  • This reduces hidden context and file scraping.
  • Say this is how agents reason over durable workflow state rather than hallucinated memory.
The prompt becomes thinner because the runtime becomes richer.

13 · organizational consequence

Once workflows are encoded explicitly, the org naturally splits into designing them and operating them

The architect function

  • Designs procedures, system boundaries, tests, triage loops, and approval points.
  • Defines what “good” looks like for the agents.
  • Critiques plans before complexity becomes durable.
  • Owns the workflow semantics, not just the code output.

The operator function

  • Works effectively inside authored workflows.
  • Investigates, validates, reviews, approves, refines UI, and handles bounded tasks.
  • Lets AI assign and execute work where the structure is already known.
  • Focuses human attention where judgment is still needed, not where typing used to be needed.

Why it belongs here

This is not the premise of the talk. It is the consequence. Once you understand nanoboss procedures as judgment-bearing workflow code, Pang's architect/operator split starts to feel natural instead of theoretical.

Org Consequence Notes

  • Now Pang fits, because the audience already understands procedures and checkpoints.
  • Present this as an implication of the architecture, not as labor-market futurism.
  • Architects design workflows; operators work effectively inside them.
This is the organizational consequence of explicit workflow design, not the starting assumption.

14 · medium-complexity software

nanoboss is also a testing ground for how medium-complexity software should be structured

Current refactor thesis

The active plan is not "split folders into packages." It is "simplify the ontology first, then extract libraries around the canonical concepts."
Nanoboss sits in a useful middle zone: roughly 22k lines of app code and about the same again in tests. Not small, not huge. Big enough that naming and boundaries matter, small enough that you can still refactor the whole thing coherently.

This matters to the talk because it reinforces the same philosophy: do not build around accidental protocol or storage terms; build around the real nouns developers need. Nanoboss is not only a testbed for agent workflow ideas. It is a testbed for how medium-complexity software should simplify itself around canonical concepts and cleaner boundaries.

Refactor Notes

  • Frame this as nanoboss being a research vehicle for medium-complexity software design under agent pressure.
  • The ontology cleanup mirrors the same judgment-boundary philosophy.
  • Do not get lost in package names; focus on cleaner canonical concepts.
This is the "nanoboss as research vehicle" part of the story, broader than agent runtime design alone.

15 · close

What nanoboss is really offering

Not "here is the agent harness you should all use."
Instead: here is a runtime where you can build the harness your workflow actually needs.

Closing points

  • Procedures are the unit of customization.
  • Sessions, runs, refs, and continuations give those procedures durable structure.
  • ACP and MCP are not the product. They are transport choices inside the runtime.
  • Better agent systems put humans at the exact points where structural judgment matters.
  • The active refactor is trying to make those runtime concepts cleaner, more portable, and more library-like.
  • The lightsaber metaphor works because nanoboss is about crafted workflow identity, not generic agent theater.
If you want one line for the presentation: nanoboss treats agentic workflow as software architecture, not as a prompt preset, and uses procedures to place human judgment exactly where it matters.

Close Notes

  • Restate the strongest line: nanoboss lets you encode where human judgment belongs.
