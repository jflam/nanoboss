ypi vs nanoboss: can ypi-style Recursive Language Models live inside nanoboss procedures?

Short answer: Yes for recursive orchestration; not yet a faithful drop-in for isolated parallel code-editing.
Core finding: your intuition is basically right. Nanoboss already has the two hardest ingredients for an RLM-style system — fresh child agent sessions and nested procedure invocation with durable run lineage. What it doesn’t yet have is ypi’s built-in per-child worktree isolation and first-class recursion guardrails (depth/calls/budget/timeout/cost). So an /rlm procedure is feasible today, but a ypi-equivalent recursive coding mode needs a small runtime layer on top of the current procedure SDK.

1. What ypi actually is

  • Architecture: a thin shell around Pi. The launcher puts rlm_query on PATH, builds a combined system prompt from SYSTEM_PROMPT.md + the live rlm_query source, and starts Pi.
  • Recursion unit: each rlm_query call spawns another Pi process with the same prompt and tools; children can recurse again until the max depth.
  • Context model: child gets piped stdin or inherited $CONTEXT; original prompt is materialized as $RLM_PROMPT_FILE.
  • Isolation: recursive children get their own jj workspace; parent worktree stays untouched until the patch is absorbed.
  • Guardrails: depth, max calls, budget, timeout, model routing, PATH scrubbing at the leaf, session tree tracing, optional async fan-out.

2. What nanoboss actually is

  • Architecture: a procedure-oriented runtime, not a single generic inner loop. Procedure code decides continuity, checkpoints, recovery, validation, and “done”.
  • Execution core: ctx.agent.run, ctx.procedures.run, ctx.session, ctx.state, ctx.ui, plus pause/resume.
  • Downstream transport: ACP over stdio; each new agent session mounts the global nanoboss MCP server automatically.
  • Durability: every top-level, nested procedure, and agent call becomes a stored run/ref tree; descendants/ancestors are queryable later.
  • Existing evidence: /research, /second-opinion, /simplify2, and /autoresearch already do multi-step orchestration, typed outputs, resumable checkpoints, and background dispatch.

3. The most important similarity — and the biggest gap

Why this can work

  • Fresh child procedures already exist. ctx.procedures.run(name, prompt, { session: "fresh" }) gives the child a private default-session binding; tests verify the root binding is untouched.
  • Fresh child agent contexts already exist. ctx.agent.run(...) defaults to fresh ACP sessions; session: "default" reuses the persistent conversation when needed.
  • Child results are first-class objects. Nanoboss stores summaries, data refs, explicit schemas, descendants, and UI events — stronger observability than ypi’s temp-file trace alone.

The blocker for a faithful ypi port

  • No per-child worktree isolation in the current SDK. Child procedures and child agent calls inherit the same cwd/worktree.
  • No built-in recursion budget API. Cancellation exists, but depth/call/budget/timeout/cost accounting must be implemented by the procedure itself.

5. Feasibility judgment

  • Analysis / summarization RLM: feasible today. A disk procedure could recursively decompose work, invoke fresh child procedures, and aggregate typed outputs.
  • Sequential coding RLM: also feasible today. It would look more like /simplify2 or /autoresearch: one worktree, explicit validation/revert/commit steps, no parallel patch branches.
  • Model-directed self-recursion: feasible if the recursive policy is exposed as a procedure (for example /rlm) and the downstream agent is taught to call it through the global MCP tools.
  • ypi-style parallel isolated subtree editing: not cleanly feasible with current stable APIs, because there is no per-child worktree/cwd override analogous to ypi’s jj workspaces.

Bottom line

Nanoboss can implement the recursive language model control pattern. It cannot yet implement the full ypi coding-agent ergonomics without adding workspace isolation and reusable recursion guardrails.

6. Recommended implementation path inside nanoboss

Phase 1 — ship an /rlm procedure

  • Encode a typed recursive state envelope: { depth, maxDepth, remainingCalls, deadlineMs, objective, contextRef?, parentRun? }.
  • Use one typed agent call to return a decomposition plan: leaf | split | stop.
  • For split, recurse via ctx.procedures.run("rlm", childPrompt, { session: "fresh" }); for leaf, call ctx.agent.run.
  • Store chunk/context payloads as refs; materialize to files only when a child tool needs filesystem access.

Phase 2 — add the runtime pieces ypi gets “for free”

  • Workspace API: per-child worktree or cwd override for agent/procedure calls.
  • Recursion guardrails helper: shared depth/call/budget/timeout/cost accounting so every recursive procedure does not reinvent it.
  • Child fan-out helper: a first-class nested async dispatch API instead of reaching into engine internals.

Phase 3 — optional ypi-style self-similarity

  • Expose /rlm through the existing global MCP path so the default agent can self-dispatch it as a tool.
  • Keep the recursive strategy prompt/spec next to the procedure so the procedure can read and evolve its own decomposition rules.

7. Why nanoboss may actually be the better product substrate

  • More durable: recursive trees become stored runs/refs, not just ephemeral child sessions.
  • More inspectable: MCP tools and run lineage make postmortems easy.
  • More controllable: typed outputs, continuations, and structured UI panels fit product workflows better than pure prompt choreography.
  • Less elegant: you lose some of ypi’s beauty — one prompt, one shell function, universal self-similarity.

Best framing: ypi is the minimal research vehicle; nanoboss is the better place to turn the idea into a durable runtime feature.
